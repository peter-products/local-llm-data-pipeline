# Local LLM Data Cleansing Pipeline

A pattern for cleansing, normalizing, and enriching scraped/structured data using a **local LLM** (Qwen 3 8B via Ollama) instead of paying per-token to a cloud provider. The example use case is a real one: turning ~40K Glassdoor employee review records into a B2B sales-intelligence dataset with normalized industries and per-record sales opportunity classifications.

## Why local LLM for this

I had a dataset of ~40K Glassdoor records that needed:
- **Industry classification** for the ~15% of companies missing one (LLM judgment based on name/domain/description)
- **GTM opportunity tagging** — does this employee complaint reveal a problem a B2B vendor could sell into?
- **Schema normalization** — flattening, renaming, adding derived fields

Doing this on a hosted LLM (Claude, GPT-4) would have cost real money per dataset and per re-run. I iterate on prompts a lot — easily 10+ runs over the full dataset before the output is right. At cloud LLM prices that's hundreds of dollars per dataset.

Running it locally on a GTX 1070 with Qwen 3 8B: **$0**, ~6 hours per full pass, fully offline, no rate limits, no terms-of-service worries about what I'm allowed to feed it.

## Architecture

The pipeline is intentionally split into discrete phases that can be run, debugged, and re-run independently:

```
raw source data
      │
      ▼
┌─────────────────┐
│ 1. Preprocess   │  Schema normalization, lookup-based enrichment,
│   (preprocess.py)│  derived fields. No LLM. Pure Python.
└────────┬────────┘
         ▼
┌─────────────────┐
│ 2. LLM Worker   │  Calls local Ollama (Qwen 3 8B) for tasks the
│  (llm_worker.py) │  lookup couldn't handle. Checkpointed every 50
│                  │  records. Validates LLM output against allowed
│                  │  enums before accepting.
└────────┬────────┘
         ▼
┌─────────────────┐
│ 3. Validate     │  Sanity checks against expected ranges, completeness
│  (validate.py)   │  reports, distribution stats. Catches drift.
└─────────────────┘
```

There's also a **lookup builder** (`build_industry_lookup.py`) — a one-time script that takes all unique industry strings from the source data and asks Qwen to map each one to a normalized category. The result is cached as `config/industry_lookup.json` and used by `preprocess.py` to skip the LLM entirely for known mappings. This is the key cost optimization: the LLM only runs on records the lookup couldn't resolve.

## Decisions worth calling out

| Decision | Why |
|---|---|
| **Local Qwen 3 8B over cloud LLMs** | Free, offline, no rate limits. Pricing for cloud LLMs only makes sense if your use case has user-facing latency requirements. Batch enrichment doesn't. |
| **`think: false` in Ollama options** | Qwen 3 has chain-of-thought reasoning enabled by default. For this kind of structured classification task, the reasoning tokens are wasted (and the JSON parsing breaks more often). Disabling thinking cut latency by ~40%. |
| **Lookup table before LLM call** | The first pass over the data builds a lookup table of all unique industry strings → normalized categories. Subsequent records hit the lookup first; only unseen strings hit the LLM. Drops total LLM calls by ~85%. |
| **Validate LLM output against allowed enum** | The LLM occasionally hallucinates new categories. If the response isn't in the allowed set, treat it as null. Don't trust the model — verify. |
| **Two retries with backoff, then null** | Don't let one bad LLM response or one network blip kill a 6-hour batch. Retry once, then write null and move on. Track null rate in validation. |
| **Checkpoint every 50 records** | If something crashes mid-run (Ollama OOM, my laptop sleeping), I can resume from the last checkpoint instead of re-running from zero. |
| **Strip markdown fences from LLM output** | Even with explicit "no markdown" instructions, the model wraps JSON in ```json fences ~10% of the time. Cheaper to handle defensively in the parser than to keep refining the prompt. |
| **Validation phase as a separate script** | Distribution checks (% of records with each category, % null, etc.) reveal prompt drift faster than spot-checking. Run after every full pass. |

## Tech stack

- **LLM**: Qwen 3 8B via [Ollama](https://ollama.ai/) (local, runs on a GTX 1070)
- **Language**: Python 3.12
- **Dependencies**: `requests` only — no Anthropic SDK, no LangChain, no ML libraries

## Project structure

```
local-llm-data-pipeline/
├── scripts/
│   ├── preprocess.py            # Phase 1: schema + lookup-based enrichment
│   ├── build_industry_lookup.py # One-time: build the industry lookup table
│   ├── llm_worker.py            # Phase 2: LLM calls for unresolved records
│   └── validate.py              # Phase 3: distribution and completeness checks
├── config/
│   └── industry_lookup.json     # Generated by build_industry_lookup.py
└── data/                        # (gitignored) raw input + cleansed output
```

## Running it

Prerequisites:
1. Install [Ollama](https://ollama.ai/) and pull the model: `ollama pull qwen3:8b`
2. Make sure Ollama is running: `ollama serve`
3. Python 3.12+ with `requests`

```bash
# put your raw input file at data/source/your_data.json
# (or override SOURCE_FILE env var)

# 1. build the lookup (one-time)
python scripts/build_industry_lookup.py

# 2. preprocess
python scripts/preprocess.py

# 3. enrich via LLM (this is the long one — checkpointed)
python scripts/llm_worker.py

# 4. validate
python scripts/validate.py
```

## What's in the code

The example pipeline is built around a Glassdoor employee review dataset, but the **pattern** is what matters. To adapt it to a different dataset:

1. Replace the `task2_prompt` and `task3_prompt` functions in `llm_worker.py` with prompts for your domain
2. Replace the `ALLOWED_INDUSTRIES` and `ALLOWED_CATEGORIES` enums with your own
3. Adjust `process_record` to map your input schema to your desired output schema
4. Update `preprocess.py` for any source-specific normalization

Everything else — the checkpointing, retry logic, JSON parsing defenses, validation phase — is reusable as-is.
